# Off-limits zones ("fake collision" walls) — Design

Date: 2026-06-11

## Summary

Add **off-limits zones** to the editor: thin, vertical, freely-oriented
rectangular **walls** that an author places to mark parts of a scene as
out-of-bounds. Multiple zones are supported, exactly like annotations, and they
are persisted in the project document.

In the exported viewer, injected companion JavaScript (the same injection
mechanism `src/viewer-companion/annotation-links.ts` already uses) reads the
camera each frame and performs an analytic **"fake collision"**: if the camera
would cross a wall, it is snapped back to the last safe position. On contact the
wall briefly appears as a semi-transparent **red** quad and a single **global
message** ("You have reached the end of the scene.") is shown, then both fade.

Blocking is done entirely by the injected companion (Approach A) — it does **not**
use the voxel collision pipeline and is independent of the collision-export
toggle.

## Goals

- Author can create, select, move, rotate (all axes), resize, and delete
  multiple off-limits walls in the editor, reusing the annotations management
  UX and the translate/rotate gizmos (walls render as thin plane meshes, not
  the axis-aligned box shader — see §3/§4).
- Walls + a single global message persist in the project document
  (round-trip through save/load), like annotations.
- The exported viewer blocks the camera from passing through any wall, shows a
  brief red flash of the wall, and displays the global message.
- Works through the existing export injection path (local and server export,
  streaming and non-streaming ZIP, and single-file HTML), since the injection
  is appended to `index.html` like the annotation-links companion.

## Non-goals (out of scope)

- Per-wall messages (a single global message is used).
- Feeding zones into the voxel collision grid (`writeVoxel`) — blocking is
  analytic, in injected JS.
- Box-volume or footprint-extrusion zone shapes (a zone is a thin wall).
- Any change to the viewer library (`@playcanvas/splat-transform`); all viewer
  behaviour is added via injected companion code.

## Open risk — viewer camera access (spike first)

The exported viewer is generated by `@playcanvas/splat-transform` and its
camera-controller code is minified. The whole feature's blocking + red-quad
rendering depends on injected JS being able to, at runtime:

1. read the active camera entity's world position each frame,
2. set the camera entity's position (to clamp it), and
3. create a temporary mesh entity in the running scene (the red quad).

Reading the camera looks reliable (the annotation-links companion already drives
the running viewer). Setting the camera and adding a scene entity are *likely*
but unverified. **Implementation step 1 is a throwaway spike** that confirms all
three against a real exported viewer. If (2)/(3) prove impossible we revisit the
approach (fall back toward voxel-based blocking for the physical block, keeping
the red/message as a DOM overlay). This risk is isolated to step 1.

## Background / current architecture

### Annotations (the management/persistence pattern we mirror)

`src/annotations.ts` is the model: it holds the live `annotations: AnnotationData[]`,
exposes events (`annotations.list / byId / selected / select / changed /
insertRaw / removeRaw / updateRaw`), and defines `Add/Remove/Update` edit ops
wired into `EditHistory` for undo/redo. `docSerialize.annotations()` /
`docDeserialize.annotations()` round-trip the array into the project
`document.json`. `src/tools/annotation-tool.ts` is a floating toolbar that edits
the selected item and drives a `TranslateGizmo`.

### Box selection / box shape (the editor rendering + gizmo we reuse)

`src/box-shape.ts` (`BoxShape extends Element`) renders a wireframe box via a
shader material and is positioned/scaled from a `pivot` entity.
`src/tools/box-selection.ts` attaches a `TranslateGizmo` to that pivot. We reuse
`BoxShape` to draw a wall as a thin box, and pair the `TranslateGizmo` with a
full `RotateGizmo` (all three axes — the PlayCanvas default, no constraint code).

### Viewer companion injection (the export hook we reuse)

`src/viewer-companion/annotation-links.ts` exposes
`buildAnnotationLinksInjection()` producing CSS + JS + a
`window.__supersplatAnnotationLinks = …` JSON blob, injected before `</body>`
of the exported `index.html` by `src/splat-export-core.ts`. The off-limits
companion follows the identical shape and is injected at the same site, so it
flows through every export variant automatically (local/server, streaming/non-
streaming ZIP, single-file HTML).

## Detailed design

### 1. Data model

```ts
type ZoneData = {
    id: string;                 // stable id for selection/undo (like annotations)
    position: [number, number, number];   // world-space center of the wall
    rotation: [number, number, number, number];  // quaternion [x, y, z, w]
    width: number;              // horizontal extent (local X)
    height: number;             // vertical extent (local Y)
};
```

Orientation is a **quaternion** (matches `entity.getRotation()` from the rotate
gizmo; avoids Euler order/gimbal ambiguity). The wall is the rectangle in the
zone's local XY plane (local Z is the wall normal); analytically it has zero
thickness, but it is drawn as a thin box in the editor and as a thin quad in the
viewer.

A single global message is stored alongside the zones as `message: string`,
where an **empty string means "use the localized default"** and any non-empty
string is custom text shown verbatim. The default is localized (see §8) — both
in the editor (placeholder text in the author's locale) and in the viewer
(by the viewer's browser language).

### 2. Editor model (`src/off-limits-zones.ts`, new)

Mirror `src/annotations.ts`:

- Live `zones: ZoneData[]` plus a `message: string`.
- Events: `offLimitsZones.list / byId(id) / selected / select(id) / changed /
  insertRaw / removeRaw / updateRaw`, and `offLimitsZones.message` (get) /
  `offLimitsZones.setMessage` (set, undoable).
- `Add / Remove / Update` edit ops (and a message-change op) registered with
  `EditHistory` for undo/redo.
- New zones default to a reasonable size (e.g. width/height ~1m) positioned at
  the current viewport focus, oriented to face the camera.

### 3. Editor tool (`src/tools/off-limits-zone-tool.ts`, new)

Mirror `src/tools/annotation-tool.ts`:

- A floating toolbar for the **selected** wall exposing numeric `width` /
  `height` inputs, plus a single global **message** text field (persisted in the
  document — see §5). The field's placeholder shows the editor-localized default;
  leaving it empty persists `""` (= use the localized default in the viewer),
  while typed text is stored as custom verbatim text.
- Render walls as thin rectangular **plane (quad) meshes** with a transparent
  material (see §4). NB: `BoxShape`'s shader (`src/shaders/box-shape-shader.ts`)
  is a world-space **axis-aligned** ray-box intersection and ignores rotation,
  so it cannot draw a rotated wall — a plane mesh (which respects the entity's
  `matrix_model`) is used instead. The translate/rotate **gizmos** are still
  reused from the box-selection/transform tools.
- Drive a `TranslateGizmo` **and** a `RotateGizmo` (all axes) bound to the
  wall's pivot. Each gizmo change commits an `UpdateOp` (position / rotation),
  mirroring how the annotation tool commits on field change.
- Creation/selection/listing reuses the annotations UX (add a zone, click to
  select, multiple managed together).

### 4. Editor visibility of all zones

All zones are visible while editing, not just the selected one.

- **Unselected** zones render as a **red semi-transparent plane (quad)** (a thin
  rectangular mesh with a transparent red material), so authors can see
  boundaries at a glance.
- The **selected** zone renders as a **brighter/more-opaque red plane** of the
  same shape and carries the translate + rotate gizmos for editing.

Both use a thin rectangular plane mesh placed in the world layer at the zone's
transform (position + quaternion) and scaled to `width × height`; the mesh
respects rotation via the standard `matrix_model` path. Selecting/deselecting
only changes the material (and attaches/detaches the gizmos).

### 5. Persistence (`docSerialize` / `docDeserialize`)

Add `document.offLimitsZones: ZoneData[]` and `document.offLimitsMessage: string`
to the document, mirroring `docSerialize.annotations()` /
`docDeserialize.annotations()` (regenerate ids if missing on load). Zones persist
regardless of the collision-export toggle.

### 6. Export injection (`src/viewer-companion/off-limits-zones.ts`, new)

`buildOffLimitsZonesInjection(zones, message)` returns the CSS + JS + a
`window.__supersplatOffLimitsZones = { zones, message }` JSON blob, in the same
shape as `buildAnnotationLinksInjection()`. `src/splat-export-core.ts` injects it
before `</body>` next to `injectAnnotationLinks` (and the server path inherits it
because the injection happens inside the shared `writeViewerCore`). When there
are no zones, no injection is emitted. `message` is passed through verbatim
(empty string included); the companion resolves an empty message to the
viewer-localized default (§8).

### 7. Viewer runtime (inside the injected JS)

On load, locate the PlayCanvas app + active camera (validated by the spike).
Each frame (app update event or `requestAnimationFrame`):

1. Read camera world position `cur`; keep `lastSafe` (previous frame's accepted
   position).
2. For each wall, transform the segment `lastSafe → cur` into the wall's **local
   space** (inverse of the wall world transform built from `position` +
   `rotation`). In local space the wall is the axis-aligned rectangle
   `|x| ≤ width/2, |y| ≤ height/2` in the `z = 0` plane.
3. If the local-Z coordinate changes sign between the two endpoints **and** the
   crossing point lies within the rectangle's extents, it's a hit.
4. On the nearest hit, set the camera position to the crossing point pulled back
   slightly along the segment toward `lastSafe` (a small epsilon), so the camera
   rests just on the safe side. Then trigger feedback.

**Feedback (invisible-until-contact):** on hit, spawn a temporary
semi-transparent **red quad** entity at the hit wall's transform (sized
`width × height`), show the global message in a DOM overlay (styled like the
annotation tooltip CSS), and fade both out after a short delay.

The geometry (segment vs oriented rectangle, world↔local transform, clamp point)
is a **pure function** in its own small module that is bundled into the injection
string, so it can be unit-tested without a viewer or GPU.

### 8. Localization of the default message

The default message is localized in two independent places, mirroring how
`annotation-links.ts` localizes its "Open link" label.

**Editor (placeholder):** add an i18next key (e.g. `offLimitsZones.defaultMessage`)
to `static/locales/*.json`, with the English value
`"You have reached the end of the scene."` and translations for the other
shipped locales. The tool's message field uses this as placeholder text in the
author's locale.

**Viewer (runtime):** the exported viewer is standalone with no access to
i18next, so the companion ships its own default-message table keyed by primary
language subtag and detects the viewer's `navigator.language` — exactly the
pattern in `annotation-links.ts` (cover the same languages: `en, de, es, fr,
ja, ko, pt, ru, zh`; fall back base-subtag → English). At display time:
`text = customMessage || defaultByLang(navigator.language)`. So an author who
leaves the message empty gets a per-viewer localized default, while custom text
is shown as-is.

## Error handling

- Injection is only emitted when zones exist; an export with no zones is byte-for-
  byte unaffected.
- The runtime companion guards on the presence of the app/camera globals and
  no-ops (logs a warning) if they are absent, so a viewer it can't drive still
  loads normally rather than throwing.
- Deserialization regenerates missing ids and ignores malformed entries, like
  annotations.

## Testing

- **Unit (editor model):** add / remove / update / select a zone; set message;
  serialize → deserialize round-trip preserves `id`, `position`, `rotation`,
  `width`, `height`, and `message`; undo/redo restores prior state. Mirrors the
  annotations model tests.
- **Unit (collision math, pure function):**
  - A segment crossing the wall plane within extents → hit, with the clamp point
    on the `lastSafe` side and within epsilon of the plane.
  - A segment crossing the plane **outside** the extents → no hit.
  - A segment that stays on one side → no hit.
  - Arbitrary wall orientation (non-axis-aligned quaternion) handled via the
    local-space transform (rotate a known axis-aligned case and confirm the same
    result).
- **Unit (default-message resolution, pure helper):** `defaultByLang('fr')`,
  `'fr-CA'` (base-subtag fallback), and an unknown language (English fallback)
  return the expected strings; a non-empty custom message wins over the default.
- **Manual (editor):** unselected zones show as red translucent rectangles;
  selecting one swaps to the box wireframe + gizmos; deselecting swaps back.
- **Manual (end-to-end):** export a viewer with one or more walls; confirm the
  camera is blocked, the wall flashes red, and the global message appears and
  fades; with the message left empty, confirm the viewer shows the localized
  default for its browser language; confirm an export with no zones is unchanged.
- **Spike (step 1, throwaway):** confirm injected JS can read the camera, set the
  camera, and add a temporary mesh entity in a real exported viewer.
